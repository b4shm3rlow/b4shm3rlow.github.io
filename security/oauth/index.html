Ludovico Guercio
2022-09-10
security

Cos’è il protocollo OAuth ? Come Funziona ?

OAuth è un protocollo di rete per l’autorizzazione progettato per lavorare con http. Consiste nel rilascio di un token di accesso da parte di un server autorizzativo ad un client di terze parti, previa approvazione dell’utente proprietario della risorsa cui si intende accedere. Tutto ciò entra in auto per le app moderne che sono progettate interno alle API. Le API forniscono accesso ai dati o servizi di applicazione o servizio.
In passato un utente doveva condividere le credenziali con l’applicazione per abilitare le chiamate API: ciò porta una quantità di accesso non necessaria e non salvaguardando la protezione dei dati di accesso.
Il protocollo fornisce oggigiorno una soluzione migliore per app e chiamate API. Infatti possiamo definirlo come un sistema di “accesso delegato”. Comunemente utilizzato come modalità per gli utenti nell’Internet per garantire alle applicazioni e siti web, l’accesso alle proprie informazioni senza fornire loro alcune password e credenziali.

OAuth 1.0

OAuth 1.0 è stata la prima versione del protocollo, ideato per autorizzare terze parti a gestire documenti privati senza condivisione di password, garantendo così l’accesso delegato ad un client specifico per determinare risorse e per un tempo limitato.
Il protocollo attualmente è stato “upgradato” perchè presentava falle di sicurezza a livello di confidenzialità: sebbene garantiva che il client non accedesse ad informazioni indisiderate, non garantiva che l’uso di risorse autorizzate rimanessero nell’ambito specifico. Infatti, OAuth suggeriva al server di proteggersi con il protocollo TLS. Per le integrità delle informazioni bisogna necessariamente utilizzare HTTPS, l’HTTP sicuro con crittografia assimentrica che ti protegge da attacchi “Man in the Middle”.
Ulteriore problema fu quello legato al “phishing”, dove si rischiava che l’utente potesse essere indirizzato ad una pagina di accesso fake del server richiendo di conseguenza l’input delle credenziali.

OAuth 2.0 (Attuale)

Con l’aggiornamento ad OAuth 2.0 si è aumentato il livello di affidabilità. Il principio di funzionamento è lo stesso, con i medesimi attori, ma presenta una chiara divisione di ruoli tra cui l’implementazione di un mediatore tra client e server.
Il mediatore ha il compito di gestire i token per l’autorizzazione per il client; il server che funge da mediatore, può essere lo stesso che ospita le risorse ai quali accedere (ad esempio, autorizzazione tramite Google e richiesta delle API Google tipo Google Maps).

OAuth fornisce una soluzione di autorizzazione, non di autenticazione!
Consente quindi ad una applicazione di effettuare chiamate API per proprio conto o per conto di un utente, con la chiamata vincolata all’ambito di una richiesta autorizzata. Il passaggio di autenticazione in OAuth 2.0 convalida che l’utente sia autorizzato a prestare consenso per autorizzare una richiesta di accesso per una specifica risorsa. Il token di accesso è destinato esclusivamente all’accesso API e non per trasmettere informazioni sull’evento di autorizzazione. Il token, intoltre, ha una durata più lunga ed estendibile.

Attori in OAuth

  • Utente (Resource Owner): persona che può dare accesso ad una porzione del proprio account. La persona può garantire l’accesso alle risorse protette nel “Resource Server”
  • Resource Server: server utilizzato per accedere alle risorse protette (Tramite servizio API)
  • Client: è l’applicazione che cerca di ottenere l’accesso all’account dell’utente, e che richiede il permesso dell’utente prima di poter accedere alle risorse protette
  • Authorization Server: server ritenuto affidabile dal Resource Server che presenta l’interfaccia dove l’utente approva o nega la richiesta di accesso. Con OAuth 2.0 può corrispondere anche al Resource Server

Tokens in OAuth

  • Authorization Code: codice intermedio opaco, resistituito ad un’app e utilizzato per ottenere un access token e un facoltativo refesh token. Ogni codice di autorizzazione può essere usato una sola volta!
  • Access Token: codice utilizzato dall’app per accedere all’API. Rappresenta l’autorizzazione dell’applicazione a chiamare l’API ed ha una scadenza
  • Refresh Token: codice opzionale che può essere utilizzato da un applicazione per richiedere un nuovo token di accesso quando il precedente è scaduto

Come vengono gestite le autorizzazioni ?

Il protocollo OAuth definisce 4 metodi con cui le applicazione ottengono le autorizzazioni a chiamare le API. Ogni metodo usa un diverso tipo di credenziali per rappresentare l’autorizzazione: questi tipi di autorizzazioni sono chiamate “Authorization Grant“.
Il tipo concessione(grant) da utilizzare dipende dal caso d’uso dell’app.

Tipi di Authorization Grant:

  • Authorization code grant
  • Implicit grant
  • Resource owner password credent grant
  • Client credential grant

Authorization Code Grant

L’Authorization Code Grant usa due tipi di richieste dall’applicazione per il server d’autorizzazione per ottenere un access token. Questo meccanismo di autorizzazione utilizza anche una chiava di prova per lo scambio chiamata “Key for Code Exchange (PKCE)“

  1. All’inizio l’utente accede all’applicazione
  2. Qui avviene poi la prima richiesta: il browser dell’utente viene reindirizzato all’endpoint di autorizzazione presso l’ Authorization Server con una richiesta per autorizzare una chiamata API per conto dell’utente.
  3. Il reindirizzamento del browser abilita l’Authorization Server ad interagire con l’utente per autenticarlo e ottenere il suo consenso per la richiesta di autorizzazione.
  4. L’utente autenticandosi concede il autorizzazione per la richiesta API al server
  5. Dopo aver ottenuto l’autorizzazione, il server reindirizza il browser dell’utente all’applicazione insieme ad un codice di autorizzazione (Authorization Code)
  6. Da qui inizia una serie di importanti scambi di comunicazioni che sono nascosti all’utente, che avvengono tra applicazione e server. L’app utilizza il codice d’autorizzazione per inviare la seconda richiesta chiamata “backchannel request” all’authorization server token endpoint per ottenere l’Access Token. In più l’app riceve un ulteriore token chiamato “Refresh Token”. Questo token server per poter richiedere una nuova chiamata API saltanto i passaggi iniziali.
  7. L’app ottenendo l’Access Token lo utilizza per richiedere l’accesso al Resource Server per l’utilizzo dell’API. Avviene in fine la convalida del Resource Server insieme all’Authorization Server che e ritorna la risposta all’applicazione, che poi fornisce le informazioni sul display dell’utente

Nel diagramma viene mostrato l’uso della chiave PKCE. PKCE può essere usato come meccanismo di authorization e token requests per garantire che l’applicazione che ha richiesto un authorizazion code è la stessa applicazione che utilizza quel codice per richiedere l’access token. In sostanza con l’uso di questa chiave si verifica anche l’autenticità dell’applicazione che userà la richiesta API.

PKCE: come avviene questa verifica?

Durante la prima richiesta, l’authorization server si fa mandare dall’applicazione questo famoso codice “Code Challenge” (creato dall’app per identificarla ed è protetto a livello crittografico); questo codice viene verificato nella fase 6 durante la richiesta dell’Access Token (lo si vede anche nel diagramma, avviene la “code verification”).

Questo passaggio è importante a livello di sicurezza, soprattutto con client pubblici e app mobile, dove si potrebbero verificare degli attacchi MITM - Man In The Middle . Gli attaccanti, qualora questo controllo fosse assente, possono intercettare l’authorization code. La protezione completa dell’attacco è garantita dalla presenza dell’authorization server che effettua i controlli dei codici di Authorization Code, di Access Token siano corretti, e che sia quella l’applicazione che li abbia richiesti.

Gli altri tipi di autorizzazione

  • Implicit Grant : sviluppato per essere usato per client pubblici come le applicazioni single page. L’uso di questo grant restituisce un token di accesso all’applicazione con una sola richiesta. Non supporta chiamate endpoint token di un authorization server; per compensare il problema, la concessione implicita fa in modo che il server di autorizzazione risponde ad un autorizzazione ritornandogli i token all’app in un URL frammentato e hashato. Inoltre, restituendo il token in un URL, espone il token nel browser in modo pericoloso.
  • Resource Owner Password Credential Grant : qui l’applicazione raccoglie direttamente le credenziali dell’utente invece che reindirizzare l’utente al server. L’app passa le credenziali raccolte all’authorization server per la convalida come parte della sua richiesta per ottenere l’access token. Questo metodo di grant è sconsigliato perchè espone le credenziali dell’utente direttamente all’applicazione
  • Client Credential Grant : è utilizzato quando un app chiama una richiesta API per accedere alle risorse che l’applicazione possiede. L’applicazione utilizza il client credential grant e si autentica al server di autorizzazione con le proprie credenziali per ottenere l’access token. Viene utilizzato dai client per accedere alle risorse su se stessi anzichè per accedere alle risorse di un utente
